# Tempo Setup and Configuration

## Overview

Tempo is a high-volume, minimal-dependency trace storage backend used for storing and querying traces in Grafana. It serves as the trace storage component in our OpenTelemetry observability stack.

## How We Set It Up

### 1. Docker Compose Integration

We added Tempo as a service in `docker-compose.yml`:

```yaml
tempo:
  image: grafana/tempo:2.10.3
  command: ["-config.file=/etc/tempo.yml", "-log.level=info"]
  ports:
    - "3200:3200"   # Tempo
    - "4317:4317"   # OTLP gRPC
    - "4318:4318"   # OTLP HTTP
  volumes:
    - ./tempo.yml:/etc/tempo.yml
  networks:
    - otel-net
  depends_on:
    - otel-collector
```

### 2. Configuration File (tempo.yml)

We created multiple iterations of `tempo.yml` to configure Tempo:

#### Initial Configuration (v1)
```yaml
server:
  http_listen_port: 3200

distributor:
  receivers:
    otlp:
      protocols:
        grpc:
          endpoint: "0.0.0.0:4317"
        http:
          endpoint: "0.0.0.0:4318"

ingester:
  trace_idle_period: 10s
  max_block_bytes: 1_000_000
  max_block_duration: 5m

compactor:
  compaction:
    compacted_block_retention: 1h

storage:
  trace:
    backend: local
    local:
      path: /tmp/tempo/traces
    wal:
      path: /tmp/tempo/wal
    block:
      bloom_filter_false_positive: .05
      index_downsample_bytes: 1000
      encoding: zstd
```

#### Alternative Configuration (v2) - Simplified
```yaml
server:
  http_listen_port: 3200

distributor:
  receivers:
    otlp:
      protocols:
        http:
          endpoint: "0.0.0.0:4318"

storage:
  trace:
    backend: local
    local:
      path: /tmp/tempo/traces
```

#### Command-Line Approach
We also tried configuring Tempo using command-line flags instead of config file:

```yaml
tempo:
  image: grafana/tempo:2.10.3
  command: [
    "-target=all",
    "-config.file=/etc/tempo.yml",
    "-distributor.receivers.otlp.protocols.http.endpoint=0.0.0.0:4318",
    "-ingester.trace_idle_period=10s",
    "-storage.trace.backend=local",
    "-storage.trace.local.path=/tmp/tempo/traces"
  ]
  # ... rest of config
```

## Issues Encountered

### Primary Issue: Kafka Dependency in Distributor Module

The main problem we've faced is that Tempo v2.10.3's distributor module requires Kafka to be configured, even when using local storage backend. This causes the service to fail on startup with the error:

```
failed to create distributor: the Kafka topic has not been configured
```

#### Root Cause
- Tempo's distributor is designed for high-scale deployments and assumes Kafka for message queuing
- Even with `backend: local`, the distributor initialization checks for Kafka configuration
- This appears to be a design decision in Tempo v2.10.3

#### Attempts to Fix
1. **Multiple Configuration Iterations**: Tried various `tempo.yml` configurations, removing distributor-specific settings, using minimal configs
2. **Command-Line Flags**: Attempted to override configuration via command-line parameters
3. **Version Investigation**: Researched if older versions (like 2.0.x) have simpler requirements, but stuck with 2.10.3 for compatibility
4. **Alternative Backends**: Considered switching to other trace storage solutions

#### Current Status
- Tempo service consistently fails to start due to the Kafka dependency
- Other services (Grafana, OTEL Collector, Prometheus) are running successfully
- Docker networking and connectivity between services is working
- Traces are being generated by the Node.js app but cannot be stored in Tempo

## Next Steps

To resolve the Tempo issue, we have several options:

1. **Switch to Jaeger Backend**: Replace Tempo with Jaeger, which has simpler setup requirements
2. **Downgrade Tempo**: Use an older version (e.g., 2.0.x) that may not require Kafka
3. **Add Kafka**: Configure a minimal Kafka setup for Tempo (complexity increase)
4. **Alternative Storage**: Use Loki or other backends for trace storage
